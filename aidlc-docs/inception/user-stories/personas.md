# Personas — ontology-agent (Mini AIP)

利用者は MCP クライアント（Claude Code/Desktop 等）経由でシステムに自然言語で問い合わせる。
各ペルソナは認証されたロールを持ち、閲覧境界（行レベル権限）が異なる。

---

## P1. CS（カスタマーサポート）担当
- **目的**: 個別の顧客・サービスに関する情報を素早く正確に把握し、問い合わせ対応する。
- **代表タスク**: 顧客の契約状況・直近の問い合わせ・障害イベントの照会。
- **クエリ傾向**: 単一オブジェクト中心（顧客X、チケットY）+ リンク探索。
- **閲覧境界**: 自分が担当するサービス／顧客の行のみ。他チームの顧客は不可。
- **アクション**: チケットのタグ付け・ステータス更新の「提案」（実行は承認フロー）。
- **重視点**: 速度、正確性、見てよいデータだけが返ること。

## P2. PM（プロダクトマネージャー）
- **目的**: プロダクトの方向性を判断するための傾向・分析を得る。
- **代表タスク**: 機能別の要望件数、問い合わせカテゴリの推移、利用状況の集計。
- **クエリ傾向**: 集計・分析（count / group by / 期間集計）。横断的。
- **閲覧境界**: 担当プロダクト群の集計データ。個人を特定する PII は最小化（集計値中心）。
- **アクション**: 原則読み取りのみ。
- **重視点**: 横断的な集計、トレンド、セグメント比較。

## P3. Sales（セールス）
- **目的**: 売り方・ターゲティングを判断する。
- **代表タスク**: 解約率の高いセグメント、アップセル候補、契約状況の分析。
- **クエリ傾向**: 集計 + セグメント条件での絞り込み。
- **閲覧境界**: 担当アカウント／テリトリの範囲。契約・商談関連は可、CS内部メモ等は不可。
- **アクション**: 原則読み取り（将来的にフォローアップ提案）。
- **重視点**: セグメント分析、解約・契約シグナル。

## P4. 運用・ガバナンス担当（Ontology Steward / Admin）
- **目的**: Mini AIP 自体を健全に運用する。Ontology の型を定義し、ロール権限を設定し、監査を確認する。
- **代表タスク**: ObjectType/LinkType/ActionType の登録・更新、ロール別権限ポリシーの設定、監査ログの閲覧。
- **クエリ傾向**: メタデータ操作（型登録）、権限設定、監査検索。
- **閲覧境界**: Ontology 定義と監査ログにアクセス可。業務データそのものは原則最小限（職務分離）。
- **アクション**: 型・権限の writeback（管理操作）。監査ログは閲覧のみ（改変不可）。
- **重視点**: ガバナンス、追跡可能性、最小権限、職務分離。
